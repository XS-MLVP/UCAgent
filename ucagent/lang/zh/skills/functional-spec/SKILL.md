# 功能规格与检测点分析指南 (Functional Spec)

本技能负责指导用户或 LLM 基于给定的测试规划和基础信息，细化出该 DUT 应该包含的具体功能检测点。

## 工作目标
你需要输出或修改文档 `{OUT}/03_{DUT}_functions_and_checks.md`，使用之前提供的验证规划骨架作为输入。如果该文件已生成了骨架，你需要**补充**里面的具体检测点。

## 功能点标签规范

你必须使用以下三级标签结构来抽象验证点：
1. **`<FG-XXX>` (Function Group)**: 大功能分类。例如 `<FG-API>` 代表环境约束，`<FG-ARITH>` 代表算术逻辑组，`<FG-COVERAGE>` 代表可达性证明功能。
2. **`<FC-XXX>` (Function Check)**: 具体功能点。从属于某个大类。例如在 `<FG-ARITH>` 下的 `<FC-ADD-OVERFLOW>`（加法溢出检查）。
3. **`<CK-XXX>` (Check Point)**: 细粒度的检测点。这是最终要被翻译为 SVA 断言的最底层节点。

## 断言推导 (Style) 标注规则

**最重要的一步！** 每一个 `<CK-XXX>` 检测点后面，必须紧跟 `(Style: X)` 标注。X 的取值为以下四者之一：

1. **`(Style: Assume)`**
   - **适用场景**：对 DUT 输入端口合法性的验证环境约束。（绝不能用于 DUT 内部节点或输出节点）。
   - **例子**：禁止非法输入状态、限定配置寄存器为稳定值等。
   - **规范后果**：后续将直接被翻译成 `assume property` 语句。

2. **`(Style: Comb)`**
   - **适用场景**：组合逻辑即时检查。（不涉及时间跨度和多周期状态转移）。通常是对无时钟设计的全加器、MUX、纯组合逻辑算子使用。
   - **规范后果**：后续将翻译成纯组合表达式断言。

3. **`(Style: Seq)`**
   - **适用场景**：涵盖多周期的时序逻辑行为。例如握手协议（`valid && !ready |=> $stable(data)`）、状态机状态跳转、计数器逻辑等。这是包含寄存器更新的设计最常用的断言。
   - **规范后果**：后续将翻译成包含蕴含操作符 `|->`、`|=>` 或 `s_eventually` 的时序断言。

4. **`(Style: Cover)`**
   - **适用场景**：证明设计的某个状态确实是"**可达且存活的**"（Liveness / Reachability）。
   - **强烈建议**：形式化极度容易出现假阳性（因为过约束导致某种状态根本达不到而返回 PASS）。所以，**对于每一个你写的状态转移或者核心功能 Seq 检测点，尽最大可能配对提供一个 Cover！**
   - **例子**：证明 FIFO 的 Full 标志可以拉高。证明计数器可以到达最大值。

**示例示范：**
```markdown
- <CK-AXI-VALID-STABLE> **(Style: Seq)** 当valid为高且ready为低时，下一拍valid应保持高电平。
- <CK-COVER-FIFO-FULL> **(Style: Cover)** 证明FIFO可以在特定时刻打满。
```

## 注意事项
- 你应当只关注“应该检查什么逻辑”，不要在这里真的写 SystemVerilog 代码（这是后来步骤的工作）。可以写类似伪代码的逻辑表达式帮助理解。
