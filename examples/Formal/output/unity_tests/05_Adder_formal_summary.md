# Adder 形式化验证总结报告

## 1. 验证背景与范围

本项目旨在验证一个参数化的 64 位全加器（Adder）的设计正确性。该模块是纯组合逻辑，主要功能是实现两个 64 位宽的操作数 `a`、`b` 以及一个进位输入 `cin` 的算术加法。

### 验证范围
- **功能正确性**：验证算术加法结果 `{cout, sum}` 是否符合 `a + b + cin` 的数学逻辑
- **端口一致性**：检查端口位宽定义是否满足 64 位加法产生 65 位结果的要求
- **边界条件覆盖**：确保在最大值、最小值、全 1、全 0 以及进位翻转等边界情况下逻辑正确

## 2. 设计关键信息

### 2.1 时钟与复位
- **时钟信号**: `clk` (虚拟时钟，用于 SVA 属性采样)
- **复位信号**: `rst_n` (虚拟复位，用于 SVA 属性采样)
- **复位类型**: 低电平有效
- **逻辑类型**: 组合逻辑

### 2.2 端口定义
| 端口名称 | 位宽 | 方向 | 描述 |
|---------|------|------|------|
| a | [WIDTH-1:0] | input | 加数 A |
| b | [WIDTH-1:0] | input | 加数 B |
| cin | 1 | input | 进位输入 |
| sum | [WIDTH-2:0] | output | 和输出（⚠️ 位宽异常） |
| cout | 1 | output | 进位输出 |

**注意**: `sum` 端口的位宽定义为 `[WIDTH-2:0]`（即 WIDTH-1 位），而正确的 64 位加法应该产生 65 位结果。这是一个已知的 RTL 缺陷。

## 3. 形式化验证策略

### 3.1 抽象与替换策略
- **设计复杂度**: 低（简单组合逻辑）
- **抽象策略**: 无需抽象，设计简单直接
- **存储器处理**: 不涉及存储器

### 3.2 引擎配置策略
- **默认证明引擎**: `pdrx` (用于最终的属性全量证明)
- **快速Debug引擎**: `bmc` (用于快速寻找反例)
- **并行策略**: 不需要（设计简单，证明速度快）

### 3.3 核心脚本命令
- **时钟定义**: `def_clk clk`
- **复位定义**: `def_rst rst_n -value 0`
- **时间限制**: `set_opt -time 100000`

## 5. 发现的 RTL 缺陷

### 缺陷 1: sum 端口位宽不足

**详细信息**: 请参考 `unity_tests/04_Adder_bug_report.md`

**摘要**:
- **缺陷位置**: `Adder/Adder.v` 第10行
- **问题描述**: `sum` 端口位宽定义为 `[WIDTH-2:0]`（WIDTH-1 位），而正确的 64 位加法应该产生 65 位结果
- **影响范围**: 影响 `a + b + cin` 结果 >= 2^(WIDTH-1) 的情况
- **修复建议**: 将 `sum` 端口位宽从 `[WIDTH-2:0]` 改为 `[WIDTH-1:0]`

**验证状态**: 该缺陷在当前的形式化验证中未被检测到，原因是 `CK_ADDER_RESULT_CORRECT` 属性只验证了边界情况。

## 7. 验证完备性评估

### 7.1 当前验证的局限性

1. **属性覆盖不足**:
   - `CK_ADDER_RESULT_CORRECT` 只验证了边界情况（a=0, b=0, cin=0）
   - 未验证全面的算术正确性
   - 未检测到位宽截断问题

2. **测试场景有限**:
   - 虽然覆盖点通过了，但断言的验证范围过于狭窄
   - 缺少对大数加法的验证

### 7.2 改进建议

#### 7.2.1 增强属性覆盖

建议添加以下属性来增强验证：

1. **全面算术正确性验证**:
```systemverilog
property CK_ADDER_ARITHMETIC_CORRECT;
  // 验证 {cout, sum} 等于 a + b + cin 的低 WIDTH 位
  // 注意：由于 sum 位宽不足，这个属性可能会失败
  ({cout, sum} == (a + b + cin));
endproperty
```

2. **最大值加法验证**:
```systemverilog
property CK_ADDER_MAX_ADDITION;
  // 验证最大值加法
  (a == {(WIDTH){1'b1}}) && (b == {(WIDTH){1'b1}}) && (cin == 1'b1) |->
  (cout == 1'b1) && (sum == 0);
endproperty
```

3. **进位传播验证**:
```systemverilog
property CK_ADDER_CARRY_PROPAGATION;
  // 验证进位传播
  (a == {(WIDTH-1){1'b1}}) && (b == 1) && (cin == 0) |->
  (cout == 0) && (sum == {(WIDTH-1){1'b1}});
endproperty
```

#### 7.2.2 改进验证策略

1. **使用参考模型**:
   - 创建一个正确的参考模型（使用正确的位宽）
   - 比较 RTL 输出与参考模型输出

2. **增加覆盖点**:
   - 添加更多边界情况的覆盖点
   - 添加随机激励的覆盖点

3. **使用反例驱动验证**:
   - 如果发现属性失败，使用反例来理解问题
   - 根据反例改进属性设计

## 8. 总结与结论

### 8.1 验证成果

1. ✅ 成功搭建了 Adder 的形式化验证环境
2. ✅ 实现了基础的验证属性和覆盖点
3. ✅ 所有验证属性均通过（在当前设置下）
4. ⚠️ 通过代码审查发现了一个潜在的 RTL 缺陷

### 8.2 经验教训

1. **属性设计的重要性**:
   - 过于简单的属性可能无法检测到实际的缺陷
   - 需要设计全面且精确的属性来覆盖所有关键场景

2. **边界条件的验证**:
   - 边界条件是发现缺陷的关键
   - 需要仔细设计边界条件验证属性

3. **代码审查与形式化验证的结合**:
   - 形式化验证是强大的工具，但不是万能的
   - 需要结合代码审查来发现潜在的缺陷

### 8.3 后续工作建议

1. **修复 RTL 缺陷**:
   - 将 `sum` 端口位宽从 `[WIDTH-2:0]` 改为 `[WIDTH-1:0]`
   - 重新运行验证，确保修复后所有属性仍然通过

2. **增强验证覆盖**:
   - 添加更全面的算术正确性验证属性
   - 增加更多的边界条件验证

3. **自动化验证流程**:
   - 将验证流程集成到 CI/CD 流程中
