# Adder 形式化验证规划文档

## 1. 验证背景与范围
本项目旨在验证一个参数化的 64 位全加器（Adder）的设计正确性。该模块是组合逻辑，主要功能是实现两个 64 位宽的操作数 `a`、`b` 以及一个进位输入 `cin` 的算术加法。

验证范围包括：
- **功能正确性**：验证算术加法结果 `{cout, sum}` 是否完全符合 `a + b + cin` 的数学逻辑。
- **端口一致性**：检查端口位宽定义是否满足 64 位加法产生 65 位结果的要求，防止数据截断。

## 2. 验证目标
- **算术一致性证明**：证明在所有可能的输入组合下，输出值与预期数学和相等。
- **边界条件覆盖**：确保在最大值、最小值、全 1、全 0 以及进位翻转等边界情况下，逻辑依然正确。
- **发现并修复 Bug**：通过形式化工具生成的反例（Counterexample），深入分析 RTL 中的逻辑缺陷或位宽定义错误。

## 3. 假设条件与约束
- **输入不受限**：输入 `a`, `b`, `cin` 可以在其定义的位宽范围内取任何值。
- **环境假设**：由于该模块为纯组合逻辑，不涉及时钟和复位，但为了形式化工具运行，可能需要在一个虚拟时钟域下进行采样。

## 4. 挑战与对策
- **位宽挑战**：64 位加法器具有巨大的输入空间（2^129 种组合），传统的仿真方法难以全覆盖。
- **对策**：使用形式化验证工具（如 JasperGold 或 Avis），利用数学证明的方法在数秒内遍历所有输入组合，确保 100% 的证明覆盖率。
- **数据截断风险**：64 位加法可能产生 65 位结果。如果输出端口位宽不足，会导致最高位丢失。
- **对策**：在 SVA 中定义 65 位的参考模型，通过断言直接对比 RTL 输出与参考模型，捕获任何潜在的截断错误。

## 6. 核心理念
形式化验证的目标是**证伪**。属性失败（Fail）并不是坏事，而是发现设计缺陷、提高代码质量的必经之路。每一个失败的断言都提供了一个精确的反例，帮助我们定位 RTL 中的深层逻辑错误。
